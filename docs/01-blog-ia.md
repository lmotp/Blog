# Blog IA

## 목적

이 문서는 현재 블로그의 전체 정보 구조를 정리한다.  
기준은 다음과 같다.

- 읽을 내용은 링크 기반으로 관리한다.
- 메모와 작업일지는 Markdown 기반 콘텐츠로 관리한다.
- 홈은 진입점 역할만 하고, 실제 탐색은 하위 페이지에서 한다.
- 사이드바는 전체 구조를 한눈에 볼 수 있는 내비게이션 역할을 한다.

## 전체 구조

```txt
/
├─ 홈
├─ 소개
├─ 읽을 내용
│  ├─ /posts/reading
│  └─ /posts/archive
├─ 메모
│  ├─ /notes/css
│  ├─ /notes/javascript
│  └─ /notes/vue
└─ 작업일지 (/posts)
   ├─ 작업내용
   │  ├─ 01-작업내용
   │  ├─ 02-작업내용
   │  └─ 03-작업내요
   └─ 다른작업
      ├─ 01-다른작업
      ├─ 02-다른작업
      └─ 03-다른작업
```

## 상위 내비게이션

왼쪽 사이드바는 다음 축으로 구성한다.

- `HOME`
- `읽을 내용`
  - `읽을 내용`
- `보관함`
- `메모`
  - `CSS 메모`
  - `JavaScript 메모`
  - `Vue 메모`
- `작업일지`
  - `작업내용`
    - `01-작업내용`
    - `02-작업내용`
    - `03-작업내요`
  - `다른작업`
    - `01-다른작업`
    - `02-다른작업`
    - `03-다른작업`
- `소개`

사이드바는 전체 구조를 보여주되, 실제 본문은 각 페이지에서 다룬다.  
`읽을 내용`과 `메모`는 아코디언 형태로 열고 닫히는 구조를 유지한다.
`작업일지`도 아코디언 형태로 열고 닫히며, 안에서는 `작업내용`과 `다른작업` 두 묶음으로 나누고 사이에 구분선을 둔다.

## 홈

### 역할

홈은 가장 단순한 시작점이다.  
복잡한 소개보다 현재 블로그의 핵심 축인 `읽을 내용`과 `메모`를 먼저 보여준다.

### 구성

- 짧은 소개 문장
- 읽을 내용 미리보기
- 메모 바로가기

### 제외 범위

- 상세한 포트폴리오 나열
- 긴 설명 섹션
- 복잡한 액션 버튼

## 소개

### 역할

개인 소개와 경력을 정리하는 페이지다.

### 구성

- 이름
- 직무 설명
- 연락처
- 한 줄 소개
- 기술 스택
- 경력

### 특징

- 블로그 콘텐츠와 분리된 자기소개 페이지로 유지한다.
- 블로그 본문 탐색의 시작점이 아니라 보조 정보 페이지로 둔다.

## 읽을 내용

### 역할

읽을 내용은 아직 읽지 않았거나, 읽고 싶거나, 보관할 링크를 모아두는 공간이다.  
이 블로그에서는 Markdown 본문이 아니라 링크 목록 중심으로 관리한다.

### 하위 구조

#### /posts/reading

- 읽을 내용 링크를 카테고리별로 보여준다.
- 각 항목은 상태 배지와 링크로 구성한다.
- `읽을 내용` 상태를 카테고리 안에서 함께 다룬다.

#### /posts/archive

- 이미 정리했거나 보관하고 싶은 링크를 모아둔다.
- `ul > li > a` 형태의 리스트 중심으로 보여준다.
- 카테고리별로 묶어서 읽기 좋게 정리한다.

### 제외 범위

- 별도 Markdown 본문 관리
- 긴 설명 카드
- 카드 그리드형 목록

## 메모

### 역할

메모는 CSS, JavaScript, Vue 같은 주제별 정리와 관찰을 담는 영역이다.  
콘텐츠 원본은 `content/notes/*.md`에서 관리한다.

### 하위 구조

#### /notes/css

- CSS 관련 개념과 실무 메모

#### /notes/javascript

- JavaScript 동작 원리와 메모

#### /notes/vue

- Vue 3 컴포지션, 컴포넌트 패턴, 상태 처리 메모

### 특징

- 각 페이지는 Markdown 본문을 렌더링한다.
- 문서형 레이아웃을 유지한다.
- 제목, 설명, 본문 흐름은 페이지별로 맞춘다.

## 작업일지

### 역할

작업일지는 프로젝트를 하면서 생긴 의사결정, 시행착오, 구현 기록을 남기는 Markdown 글 영역이다.

### 형태

- `content/posts/*.md`에 작성한다.
- `/posts`에서 목록으로 진입한다.
- `/posts`는 작업일지 진입점이고, 안에서 `작업내용`과 `다른작업`으로 나눈다.
- `/posts/[slug]`에서 렌더링한다.
- 하나의 글은 하나의 작업 흐름이나 결정 과정을 담는다.

### 포함 내용

- 기능 추가 배경
- 구조 변경 이유
- 문제 해결 과정
- 배포/운영 관련 기록

## 상세 페이지

### 글 상세

- `/posts/[slug]`
- `content/posts/*.md`를 렌더링한다.
- 제목, 설명, 날짜, 본문을 보여준다.

### 메모 상세

- `/notes/[slug]`
- `content/notes/*.md`를 렌더링한다.
- 제목, 설명, 날짜, 본문을 보여준다.

## 콘텐츠 원칙

- 작업일지와 메모는 Markdown 원본으로 관리한다.
- 읽을 내용과 보관함은 링크 기반으로 관리한다.
- 홈은 콘텐츠 요약만 보여주고, 원본은 각 하위 페이지에서 본다.
- 새로운 페이지를 추가할 때는 먼저 IA 축에 맞는지 확인한다.

## 비주얼 원칙

- 문서형, 리스트형 UI를 우선한다.
- 카드와 박스는 최소화한다.
- border, shadow, 과한 라운드는 줄인다.
- 사이드바는 단순한 내비게이션처럼 유지한다.
